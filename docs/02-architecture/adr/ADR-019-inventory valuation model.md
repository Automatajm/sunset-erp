# ADR-019: Inventory Valuation Model

**Status:** Accepted  
**Date:** 2026-03-31  
**Deciders:** Juan (FP&A / Systems Engineering), Sunset ERP Architecture  
**Tags:** inventory, valuation, accounting, costing, average-cost, cogs, balance-sheet, uom, financial

---

## CONTEXT

ADR-014 established that `purchaseUom` is the financial unit of record for all monetary calculations. This ADR extends that rule into a complete inventory valuation model covering:

1. How unit cost is determined and updated at each GRN receipt
2. How inventory value is reported on the balance sheet
3. How COGS is calculated when goods are issued to production, sales, or adjustments
4. How the triple UOM system interacts with financial calculations without introducing rounding errors
5. Which valuation method is used and why

The system currently stores `Stock.unitCost` but has no formal definition of how it is calculated, in which unit it is expressed, or how it flows into the general ledger. This creates three risks:

**Risk 1 — Valuation inconsistency:** If different modules compute unit cost in different units (one in KG, another in SACOS), the inventory value on the balance sheet will be wrong.

**Risk 2 — COGS errors:** When production consumes materials in consumptionUom, the cost must be converted correctly to purchaseUom before calculating the JE debit to WIP. Without a formal rule, each developer implements this differently.

**Risk 3 — Average cost drift:** Without a documented recalculation formula, weighted average cost updates at GRN receipt may use inconsistent quantity bases, causing systematic over- or under-valuation.

---

## DECISION

Adopt **Weighted Average Cost (WAC)** as the single inventory valuation method for all stockable items. All cost calculations are anchored to `purchaseUom`. The `UomConversionService.calcFinancialValue()` method (ADR-014) is the exclusive entry point for all monetary conversions.

### Valuation Method: Weighted Average Cost (WAC)

WAC recalculates unit cost at every GRN receipt using the perpetual average formula:

```
newUnitCost = (existingValue + incomingValue) / (existingPurchaseQty + incomingPurchaseQty)

Where:
  existingValue        = Stock.purchaseQty × Stock.unitCost
  incomingValue        = GRNLine.purchaseQty × GRNLine.unitCost
  existingPurchaseQty  = Stock.purchaseQty  (in purchaseUom)
  incomingPurchaseQty  = GRNLine.purchaseQty (in purchaseUom)
```

**All quantities in purchaseUom. No exceptions.**

#### Example: WAC recalculation at GRN receipt

```
Item: Harina de Trigo (purchaseUom = SACO, 1 SACO = 50 KG)

Existing stock:
  purchaseQty = 20 SACOS
  unitCost    = $42.00/SACO
  totalValue  = $840.00

Incoming GRN (10 SACOS @ $45.00):
  incomingValue = 10 × $45.00 = $450.00

New WAC:
  newUnitCost = ($840.00 + $450.00) / (20 + 10)
              = $1,290.00 / 30
              = $43.00/SACO

Stock after receipt:
  purchaseQty = 30 SACOS
  unitCost    = $43.00/SACO   ← updated
  totalValue  = $1,290.00
  
  storageQty     = 1,500 KG   (30 × 50)    ← derived, not stored
  consumptionQty = 1,500 KG                ← derived, not stored
  
  unitCostStorage     = $43.00/50 = $0.86/KG  ← derived on demand
  unitCostConsumption = $43.00/50 = $0.86/KG  ← derived on demand
```

### Cost Flow at Each Transaction Type

#### 1. GRN Receipt (stock IN)

```
Source of cost: GRNLine.unitCost (in purchaseUom)
  └─ Primary: AP Invoice unit price (after 3-way match)
  └─ Fallback: PO line unit price
  └─ Fallback: Item.standardCost
  └─ Last resort: $0 (flagged for review)

Action:
  1. Calculate WAC (formula above)
  2. Update Stock.unitCost ← newWAC
  3. Update Stock.purchaseQty += incomingPurchaseQty
  4. Update Stock.storageQty += incomingStorageQty
  5. Update Stock.consumptionQty += incomingConsumptionQty
  6. Create StockMovement (all 3 qtys + unitCost in purchaseUom)

JE (auto-generated by AutomationEngine):
  DR  Inventory / Raw Materials (1300)   incomingValue
  CR  Accounts Payable (2100)            incomingValue
```

#### 2. Production Issue (stock OUT — MO material consumption)

```
Source of cost: Stock.unitCost (WAC, in purchaseUom)

Input: consumptionQty in consumptionUom (from BOM explosion)
  → Convert to purchaseQty: consumptionQty / item.consumptionFactor
  → Calculate value: purchaseQty × Stock.unitCost

Example:
  BOM requires 75 KG (consumptionUom)
  consumptionFactor = 50 (1 SACO = 50 KG)
  purchaseQty = 75 / 50 = 1.5 SACOS
  value = 1.5 × $43.00 = $64.50

Action:
  1. Convert consumptionQty → purchaseQty
  2. Deduct from Stock (all 3 qtys proportionally)
  3. Record StockMovement

JE:
  DR  WIP / Work In Progress (1410)      $64.50
  CR  Inventory / Raw Materials (1300)   $64.50
```

#### 3. Sales Shipment (stock OUT — SO fulfillment)

```
Same conversion path as production issue.
Source of cost: Stock.unitCost (WAC, in purchaseUom)
COGS = shippedPurchaseQty × Stock.unitCost

JE:
  DR  Cost of Goods Sold (5100)          COGS amount
  CR  Inventory / Finished Goods (1320)  COGS amount
```

#### 4. Inventory Adjustment (positive or negative)

```
Positive adjustment (physical count > system):
  Cost = Stock.unitCost × adjustedPurchaseQty  (at current WAC)
  JE:
    DR  Inventory (1300)                  value
    CR  Inventory Adjustment (5900)       value

Negative adjustment:
  JE:
    DR  Inventory Adjustment (5900)       value
    CR  Inventory (1300)                  value
```

#### 5. Transfer between warehouses (no cost impact)

```
Cost does not change on warehouse transfer.
Stock.unitCost is item-level, not warehouse-level.
StockMovement records the transfer with unitCost for audit trail.
No JE generated.
```

#### 6. GRN Cancellation (stock reversal)

```
When a GRN is cancelled, the WAC must be reversed:

reversalValue = cancelledPurchaseQty × originalUnitCost
  (use the unitCost at time of original receipt, stored in StockMovement)

newUnitCost = (currentTotalValue - reversalValue) / (currentPurchaseQty - cancelledPurchaseQty)

If currentPurchaseQty - cancelledPurchaseQty = 0 → unitCost = 0, flag for review
```

### Balance Sheet Reporting

```
Inventory value per item:
  totalValue = Stock.purchaseQty × Stock.unitCost

Inventory value per warehouse:
  totalValue = Σ (Stock.purchaseQty × Stock.unitCost) WHERE warehouseId = X

Total inventory (all items, all warehouses):
  totalValue = Σ (Stock.purchaseQty × Stock.unitCost) WHERE tenantId = X
```

The balance sheet always uses `purchaseQty × unitCost`. storageQty and consumptionQty are never used for financial reporting.

#### Display to warehouse manager

```typescript
// At query time — never stored
const unitCostStorage     = stock.unitCost / item.storageFactor;
const unitCostConsumption = stock.unitCost / item.consumptionFactor;
const totalValueStorage   = stock.storageQty * unitCostStorage;
```

### Schema Additions

#### `Stock` — clarify unitCost semantics

No new fields. Existing `unitCost` is formally declared as **cost per purchaseUom unit (WAC)**. Add a comment to the Prisma schema:

```prisma
model Stock {
  // unitCost: Weighted Average Cost per purchaseUom unit.
  // All inventory valuation, COGS, and JE amounts derive from this field.
  // Never stored in storageUom or consumptionUom.
  unitCost Decimal? @map("unit_cost") @db.Decimal(15, 4)
}
```

#### `StockMovement` — add financial fields

```prisma
model StockMovement {
  // existing: quantity (storageQty), uom (storageUom)
  // new:
  unitCostAtMovement  Decimal? @map("unit_cost_at_movement") @db.Decimal(15,4)
  // WAC at time of movement — required for GRN cancellation reversal
  movementValue       Decimal? @map("movement_value") @db.Decimal(15,2)
  // purchaseQty × unitCostAtMovement — pre-calculated for audit trail
}
```

#### `GoodsReceiptLine` — clarify unitCost semantics

```prisma
model GoodsReceiptLine {
  // unitCost: cost per purchaseUom unit (from PO price or AP Invoice price)
  // This is the cost used to update WAC, not a cost per storageUom
  unitCost Decimal? @map("unit_cost") @db.Decimal(15, 4)
}
```

### Valuation Rules Summary

| Scenario | Unit basis | Formula |
|----------|-----------|---------|
| Stock.unitCost | purchaseUom | WAC = (existingValue + incomingValue) / totalPurchaseQty |
| GRN receipt cost | purchaseUom | GRNLine.unitCost (from PO/AP Invoice) |
| Production COGS | purchaseUom | consumptionQty / factor × WAC |
| Sales COGS | purchaseUom | salesQty / factor × WAC |
| Balance sheet value | purchaseUom | purchaseQty × WAC |
| Display to warehouse | storageUom | storageQty × (WAC / storageFactor) — derived |
| Display to production | consumptionUom | consumptionQty × (WAC / consumptionFactor) — derived |
| 3-way match price | purchaseUom | abs(invoicePrice - poPrice) / poPrice |

## CONSEQUENCES

### Positive

- Single source of financial truth: all monetary values trace back to purchaseUom × WAC
- Balance sheet inventory value is always auditable — traces directly to GRN receipts and AP Invoices
- COGS is deterministic — same inputs always produce same output
- No rounding accumulation from repeated unit conversions — conversion happens once, at value calculation
- WAC is the simplest perpetual costing method — widely understood by accountants and auditors
- Items with factor = 1 (same unit everywhere) work identically — no special cases
- storageUom and consumptionUom enrich the user experience without contaminating financial data

### Negative

- WAC smooths cost over time — cannot identify cost of a specific lot (requires FIFO/LIFO for lot-level costing)
- GRN cancellation WAC reversal is complex when stock has been partially consumed between receipt and cancellation
- Items with no purchaseUomId configured fall back to baseUom — unit cost semantics are ambiguous until UOM is configured
- `unitCostStorage` and `unitCostConsumption` are never stored — every display query must compute them via join + division

### Neutral

- FIFO and LIFO are explicitly out of scope for Sprint 14 — WAC is the only supported method
- Standard cost (`Item.standardCost`) remains available for BOM costing and variance analysis but does not drive Stock.unitCost
- Tax calculations (VAT, import duties) are not part of WAC — handled separately in AP Invoice tax fields

## ALTERNATIVES CONSIDERED

### Alternative 1: FIFO (First In, First Out)

**Description:** Each GRN receipt creates a cost layer. Inventory is consumed from the oldest layer first. COGS reflects the oldest purchase price.

**Pros:**
- Matches physical flow for perishable goods
- Provides lot-level cost traceability
- Better for tax purposes in inflationary environments (lower COGS = higher taxable income)

**Cons:**
- Requires cost layers per (item, warehouse, lot) — significantly more complex schema
- Cannot be expressed as a single `unitCost` field — need a cost stack
- WAC is more appropriate for fungible bulk materials (flour, resin, chemicals) which are Sunset ERP's primary use case

**Why not chosen:** WAC covers 90% of manufacturing use cases. FIFO is documented as a future enhancement for food/pharma tenants with lot tracking requirements.

### Alternative 2: Standard Cost

**Description:** Each item has a fixed `standardCost`. All stock movements use this fixed cost. Variances are booked to a variance account.

**Pros:**
- Simple — no recalculation at receipt
- Variance analysis built-in
- Budgeting and cost accounting friendly

**Cons:**
- Standard costs go stale — require periodic revision
- Inventory value on balance sheet may differ materially from actual purchase cost
- Not GAAP/IFRS compliant as the primary valuation method for external reporting

**Why not chosen:** Standard cost is retained as an input for BOM costing and MRP planning, but actual inventory is valued at WAC per GAAP/IFRS requirements.

### Alternative 3: Specific Identification

**Description:** Each unit of stock is tracked to its specific purchase cost via lot or serial number.

**Pros:**
- Most accurate — COGS reflects the exact cost of the specific item sold
- Required for high-value, low-volume items (jewelry, equipment)

**Cons:**
- Only practical with lot or serial tracking enabled
- Does not work for bulk fungible materials

**Why not chosen:** Applicable only to items with `isLotTracked = true` or `isSerialTracked = true`. Documented as a future enhancement — when lot tracking is fully operational, specific identification can be enabled per item.

### Alternative 4: Store unitCost in all three UOMs

**Description:** Store `unitCostPurchase`, `unitCostStorage`, `unitCostConsumption` as separate fields on Stock.

**Pros:**
- No derivation needed at query time
- Each domain sees its own cost directly

**Cons:**
- Three fields always drift — any update must write all three atomically
- Storage cost: 3× the fields for data that is fully derivable
- When conversionFactor is updated, all three stored costs become stale simultaneously

**Why not chosen:** Derived fields should not be stored when the derivation is cheap (one division). The single `unitCost` in purchaseUom is the source of truth; the others are display conveniences computed at query time.

## IMPLEMENTATION NOTES

### WAC update trigger
WAC must be recalculated on every stock-in event:
- GRN receipt (primary)
- Positive inventory adjustment
- Warehouse transfer IN (no cost change — copy source WAC)
- Production order output (finished goods receipt — cost = sum of component WAC)

WAC must NOT be recalculated on stock-out events (issue, shipment, negative adjustment) — WAC at time of deduction is used for COGS, not recalculated.

### GRN cancellation with partial consumption
```typescript
async cancelGrn(grnId: string) {
  // Get original movement cost from StockMovement.unitCostAtMovement
  const movement = await this.getOriginalMovement(grnId);
  
  const currentStock = await this.getStock(itemId, warehouseId);
  
  if (currentStock.purchaseQty < movement.purchaseQty) {
    // Stock partially consumed — cannot fully reverse
    // Options: (1) reverse what remains, flag shortfall (2) block cancellation
    throw new ConflictException('Stock partially consumed. Cannot fully reverse GRN.');
  }
  
  // Full reversal
  const newTotalValue = (currentStock.purchaseQty * currentStock.unitCost)
    - (movement.purchaseQty * movement.unitCostAtMovement);
  const newPurchaseQty = currentStock.purchaseQty - movement.purchaseQty;
  const newWAC = newPurchaseQty > 0 ? newTotalValue / newPurchaseQty : 0;
  
  await this.updateStock({ purchaseQty: newPurchaseQty, unitCost: newWAC });
}
```

### Items without purchaseUomId configured
Until an item's UOM triple is configured, `storageFactor` and `consumptionFactor` default to 1. This means:
- `unitCost` = cost per baseUom unit
- All three UOM costs are equal
- Financially correct — no conversion error possible with factor = 1
