## Documentation Sources

| Source | Used For |
|--------|----------|
| `schema.prisma` | ERD, data models, field definitions |
| `app.module.ts` | Module dependency map |
| `*.controller.ts` | API endpoints, HTTP methods, parameters |
| `*.service.ts` | Business logic documentation |
| `dto/*.ts` | API contracts, request/response shapes |
| `git log --oneline` | Sprint history, commit references |
| Conversation history | ADRs, design decisions, rationale |
| Frontend pages | User guide screenshots and flows |

---

## ADR Format

Each Architecture Decision Record follows this structure:

```markdown
# ADR-XXX: Title

**Date:** YYYY-MM-DD
**Status:** Accepted | Superseded | Deprecated
**Sprint:** Sprint N

## Context
What problem were we solving?

## Decision
What did we decide?

## Rationale
Why this approach over alternatives?

## Consequences
What are the trade-offs?

## Alternatives Considered
What else was evaluated?
```

## Incremental Documentation Rule

> Documentation is written at the end of each sprint, not as a catch-up effort.

**After each sprint:**
1. Update `docs/03-modules/` for any modified or new module
2. Create `docs/04-sprints/sprint-XX.md`
3. Create ADR(s) for any architectural decisions made
4. Update `docs/02-architecture/diagrams/` if data model changed
5. Update user guides if new user-facing features were added

## How to Generate Documentation with Claude

To generate documentation for a module, provide Claude with:

```
1. The controller file: backend/src/modules/{module}/{module}.controller.ts
2. The service file:    backend/src/modules/{module}/{module}.service.ts
3. All DTO files:       backend/src/modules/{module}/dto/*.ts
4. Relevant schema:     sections of backend/prisma/schema.prisma
```

Claude will generate:
- API reference (endpoints, parameters, responses)
- Business logic description
- Sequence diagrams in Mermaid format
- ADR if architectural decisions are present
